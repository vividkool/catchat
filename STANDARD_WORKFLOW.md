# 標準開発ワークフロー V2 (Standard Development Workflow)
<!-- Last synchronized: 2026-04-23 -->

このドキュメントは、AI共創時代におけるプロジェクトの安全性と開発効率を両立させるための「鉄則」を定義します。

## 📌 このワークフローの3つの柱

### 🔐 **セキュリティファースト**
- すべての push を自動スキャン（Gitleaks/TruffleHog）
- 秘密情報（APIキー、JSON鍵）の混入を防止
- **秘密鍵（service_account.json等）はローカルのプロジェクトフォルダ内に置かない**
- GitHub Secrets による一元管理で、機密情報をコードから完全分離

### 💚 **エコノミー（Quota削減）**
- **Issue-Driven AI Deployment (IDD)**: すべての変更・デプロイは Issue から開始
- **Push トリガーの廃止**: `on: push` による自動ビルドを全廃し、Quota 節約と安全性を確保
- **自動プレビュー URL 通知**: デプロイ完了後、Issue にプレビュー URL を自動コメント

### ✅ **品質保証**
- 本番デプロイは **必ず人間による手動実行**
- develop でのプレビュー確認を必須化
- 自動化と人間による最終確認のバランス

---

## 1. ブランチ戦略 (Branch Strategy)

```
main (Production)
  └─ 本番環境（Firebase Hosting）
  └─ 更新は develop からの PR マージのみ
  └─ デプロイは手動（firebase deploy）

develop (Preview)
  └─ Firebase Hosting develop チャネル
  └─ Issue トリガーで自動デプロイ
  └─ ユーザーが確認してから main へマージ
```

## 2. 開発サイクル (AI共創フロー)

### STEP 1: タスクの投稿
GitHub Issue を作成し、AIへの指示書とします。
タイトルに日付（例: `2026-04-23`）を含めることで、自動実装トリガーを起動させます。

### STEP 2: セキュリティ検閲 ← **自動**
すべての push が自動スキャンされ、秘密情報の混入を未然に防ぎます。

### STEP 3: AIによる自動実装と develop デプロイ ← **自動**
Issue 投稿をトリガーとして、GitHub Actions が起動し、`todo.md` の更新とコード実装、develop へのデプロイを行います。

### STEP 4: プレビュー環境での確認と本番マージ ← **手動**
プレビュー URL で動作を確認し、問題なければ `main` へ PR マージします。

### STEP 5: 本番環境へのデプロイ ← **手動**
`firebase deploy --only hosting` を実行して本番環境へ反映します。

---

## 3. 運用の要点

### 📊 情報の集約
AIが文脈を維持できるよう、以下のファイルを最新に保ちます。
- `todo.md`: 現在の進行中タスク、完了予定
- `README.md`: プロジェクトの全体像

### 💻 複数デバイスでの作業
- 作業開始前に必ず `git pull` を行い、`todo.md` で最新状態を確認すること。
- 作業終了時は必ず `git push` し、進捗を `todo.md` に反映させること。

### 🚀 パフォーマンス第一
UI レスポンスを損なうような重い処理は避け、必要に応じて非同期処理や外部サービスを活用してください。

---

このワークフローは、**「AIが自律的に動きつつも、人間が最後の手綱を握り、機密情報を徹底的に守る」**ための標準プロトコルです。
